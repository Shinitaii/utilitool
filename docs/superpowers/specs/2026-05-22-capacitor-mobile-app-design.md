# Capacitor Mobile App — Design Spec
**Date**: 2026-05-22  
**Status**: Approved

---

## Context

Utilitool's web admin UI is desktop-centric (table layouts, sidebar, multi-panel). Meter readers need a focused mobile experience to capture readings in the field. This spec covers a standalone Capacitor mobile app targeting Android first (iOS later), living in the same monorepo under `mobile/`. The feature scope is: authentication, readings capture, reading history, and billings view — nothing else. The existing `assistant` role gates mobile users from admin-only features.

---

## Section 1: Architecture

Three areas of change:

```
new-utility-calculator/
├── api/functions/     — add POST /users + extend requireRole on readings writes
├── ui/                — extend settings create-user form with role selector
└── mobile/            — NEW standalone Svelte 5 + Vite + Capacitor app
```

**Why plain Svelte + Vite (not SvelteKit)?**  
Capacitor requires static file output served via `file://`. SvelteKit's SSR model fights this — you must prerender every route and avoid all server-only features. Plain Svelte + Vite produces a static SPA with zero friction. SvelteKit adds complexity for no gain in a 5-screen app.

**Critical files:**
- `mobile/capacitor.config.ts` — app ID, web dir, plugins
- `mobile/vite.config.ts` — build output to `dist/`
- `mobile/src/App.svelte` — hash-based router
- `mobile/android/` — generated by Capacitor CLI

## Section 2: API Changes

**File:** `api/functions/src/features/reading/reading.route.ts`

Add `'assistant'` to `requireRole` on three endpoints:
```
POST /readings          → requireRole('admin', 'landlord', 'assistant')
POST /readings/batch    → requireRole('admin', 'landlord', 'assistant')
POST /readings/ocr      → requireRole('admin', 'landlord', 'assistant')
```
GET routes stay unchanged (already open to all authenticated users).

**New endpoint:** `POST /users` (admin only)

Pre-seeds a Firestore user profile with the desired role before the user's first login. Without this, all new users default to `landlord` role on first `GET /auth/me`.

```
POST /users
Auth: requireRole('admin')
Body: { uid: string, role: 'admin' | 'landlord' | 'assistant' }
Response: 201 — the created User document (same shape as GET /auth/me)
```

New files to create (following existing feature pattern):
- `api/functions/src/features/user/user.dto.ts`
- `api/functions/src/features/user/user.controller.ts`
- `api/functions/src/features/user/user.route.ts`

Reuses existing `auth.repository.ts` for Firestore writes to the `users` collection.

## Section 3: Settings UI Changes

**File:** `ui/src/routes/(app)/settings/users/+page.svelte`

Extend existing create-user form:
1. Add **Role** select field: `Admin | Landlord | Assistant` (default: `Assistant`)
2. After `createUserWithEmailAndPassword()` succeeds, call `POST /users` with `{ uid: newUser.uid, role }` using the admin's token
3. Show error if the `POST /users` call fails (role not set — user should be deleted or retried)

    ├── lib/
    │   ├── api/

**Navigation:** Bottom nav (Home / History / Billings) visible on all screens except Login and CaptureReadings. CaptureReadings is a full-screen flow with its own back button.

**Routing:** Hash-based (`location.hash`, `hashchange` event) — no router library. 5 screens, trivial to hand-roll.

## Section 5: Mobile Readings Capture Flow

Full-screen flow, three steps:

### Step 1 — Session Setup
- Meter group dropdown (loads `GET /meter-groups?minimal=true` on mount)
- Date picker (defaults to today)
- "Start Session" CTA → loads properties and navigates to Step 2

### Step 2 — Property Cards (scrollable)
Loads `GET /properties?meterGroupId=X&limit=100`.

Each property card contains:
- Property name (prominent)
- **Camera button** → opens native camera via `@capacitor/camera` (Camera.getPhoto)
- **Photo thumbnail** (if photo taken) — tapping opens full-screen interactive preview
- Reading amount number input (manual entry, always primary)
- Status indicator: idle / photo taken / filled

**Interactive photo preview (full-screen overlay):**
- Touch gestures: pinch-to-zoom, pan, rotate
- Close button returns to card
- Photo is a local data URL only — never uploaded, never sent to the API

**No OCR, no Suggest button.** Camera is a visual aid for reading the meter; the number is always entered manually.

Floating **"Submit All" button** — enabled only when every card has a `reading_amount`.

### Step 3 — Confirmation
- Summary: meter group, date, N properties
- "Confirm & Submit" → `POST /readings/batch`
- Payload per row: `{ meter_group_id, property_id, reading_amount, reading_date }`
- `image_url` is NOT included (no Firebase Storage configured)
- On success: success toast → navigate to Home
- On error: show which rows failed, allow retry

## Admin: Interactive Image Preview

**File:** `ui/src/routes/(app)/readings/+page.svelte`

The existing readings page shows a Photo column. Extend the photo thumbnail click to open the same full-screen interactive preview (pinch-to-zoom, pan, rotate) as the mobile app. Same for any other place in the admin UI where meter images appear.

**Gemini / OCR gating:** Remove the Suggest button from the admin readings form entirely for now. OCR is not functional in the current environment (no Gemini API key configured on the backend), and the frontend has no reliable way to detect this. The button can be restored when OCR is properly set up end-to-end.

## Verification

1. **API role change**: Create an `assistant` user → log in → `POST /readings/batch` returns 201 (not 403)
2. **POST /users**: Admin creates user with role `assistant` via settings → new user logs in → `GET /auth/me` returns `role: 'assistant'`
3. **Settings UI**: Role selector appears, defaults to Assistant, saves correctly
4. **Mobile build**: `cd mobile && npm run build && npx cap sync android && npx cap open android` → app launches in Android Studio emulator
5. **Capture flow**: Select meter group → select date → fill all readings → submit → readings appear in admin readings list
6. **Camera + preview**: Photo taken → thumbnail shown → tap → full-screen with pinch/zoom/pan/rotate
7. **Admin preview**: Click photo in readings table → full-screen interactive preview opens
